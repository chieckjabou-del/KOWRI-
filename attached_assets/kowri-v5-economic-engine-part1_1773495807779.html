Règle de calcul — Bigint, jamais float

Tous les montants en unité minimale (XOF centimes ou XOF entier selon la devise). commission = gross_payout × rate_bps / 10000 avec division entière (floor). Si l'utilisateur a un abonnement actif (M03), le taux est réduit du discount_for_sub en bps. La commission est toujours écrite via Fee Engine → PLATFORM_FEES (jamais de déduction silencieuse).

Règle Wallet — Invariant absolu (aucune table de solde)

-- wallet_balance = projection des entrées Ledger. Toujours calculé, jamais stocké.

WalletView get_wallet(user_id):
  available  = ledger.get_balance("CLIENT_WALLET", user_id, status=COMMITTED)
  reserved   = ledger.sum_pending_external(user_id)   -- MOMO in-flight
  in_coffre  = ledger.get_balance("CLIENT_COFFRE", user_id)  -- M20
  RETURN { available, reserved, in_coffre, total: available+reserved+in_coffre }
  -- Aucune table wallet_balances. GET_BALANCE lit LEDGER_ENTRIES.

M01 — Commission Plateforme
Intercepte chaque payout tontine, calcule et écrit la commission via Fee Engine
2 tables · L1

Ce module s'insère entre le State Engine (signal PAYOUT_READY) et l'écriture Ledger finale. Il consulte commission_tiers, calcule en bigint, puis transmet au Fee Engine qui écrit de façon atomique.

1
Signal PAYOUT_READY depuis State Engine
Le cycle tontine est terminé. L'État Engine émet un événement. M01 est en écoute. Aucune écriture encore.
2
Calcul commission depuis commission_tiers
Lookup tranche selon gross_payout. Vérification abonnement actif (M03). Calcul bigint. Enregistrement dans commission_calculations (état PRE_COMMIT).
→ commission_amount calculé
3
Fee Engine écrit dans le Ledger (atomique)
Une seule transaction DB : DEBIT CLIENT_TONTINE_POOL (gross), CREDIT CLIENT_WALLET[bénéficiaire] (net), CREDIT PLATFORM_FEES (commission). Équilibre vérifié avant COMMIT.
→ LEDGER_COMMITTED · double-entry ✓
4
Mise à jour commission_calculations → COMMITTED
Référence du ledger_tx_id stockée dans calc. Audit log PAYOUT_WITH_COMMISSION. Notification bénéficiaire avec montant net.

Snapshots = optimisation seulement

Les snapshots quotidiens dans wallet_balance_snapshots accélèrent l'affichage mais ne font jamais autorité. En cas de divergence (divergence_flag = true), c'est toujours GET_BALANCE() depuis le Ledger qui est utilisé. Le checksum compare le snapshot au calcul Ledger pour détecter les inconsistances.

M08 — Commission Revente Position
3% acheteur + 0.5% organisateur · Tout via Fee Engine
1 table · L2
-- Flux financier revente position (atomique dans une transaction DB)
SALE_PRICE       = 140_000 XOF
PLATFORM_COMM    = 140_000 × 300 / 10_000 = 4_200 XOF   (3%)
ORGANIZER_COMM   = 140_000 × 50  / 10_000 = 700 XOF    (0.5%)
NET_TO_SELLER    = 140_000 - 4_200 - 700   = 135_100 XOF

-- Écriture Ledger double-entry (4 lignes, toutes dans le même tx DB) :
DEBIT  CLIENT_WALLET[buyer]           140_000 XOF
CREDIT CLIENT_WALLET[seller]          135_100 XOF
CREDIT PLATFORM_FEES                    4_200 XOF
CREDIT CLIENT_WALLET[organizer]           700 XOF
-- 140_000 = 135_100 + 4_200 + 700 ✓ (équilibré)

-- Position transférée logiquement dans tontine_members (hors ledger) :
UPDATE tontine_members SET user_id = buyer_id WHERE slot = sold_slot

Interaction Treasury Engine V4.0

M14 soumet une demande de réservation au Treasury Engine qui vérifie le ratio RESERVE_GUARANTY / encours crédits (invariant I6). Si le ratio reste ≥ 8%, la garantie est accordée et une portion de RESERVE_GUARANTY est allouée. L'activation sur défaut est déclenchée par le Treasury Engine selon la procédure V4.1 — M14 ne fait que demander.
